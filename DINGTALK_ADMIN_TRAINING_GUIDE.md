# Nanobot 游戏客服管理端培训手册

本文档面向两类读者：

- 产品、运营、客服主管：用于理解本系统的产品价值、管理价值和落地边界
- 钉钉管理员、人工客服、培训讲师：用于掌握如何通过钉钉与系统交互，并完成日常管理动作

本文档基于当前项目代码和现有文档整理，适用于 [nanobot-game-customer-service](F:\project\nanobot-customer-service\nanobot-game-customer-service) 当前版本。

## 1. 产品定位

本产品是一套面向游戏场景的智能客服与管理协同系统。它不是单纯的聊天机器人，也不是单纯的工单系统，而是将三类能力整合到一起：

- 面向玩家的 SOP 式自动接待
- 面向知识问题的知识库检索与 AI 辅助回答
- 面向运营和客服主管的钉钉管理端协作

系统的核心目标是：

- 提高首响速度，减少人工重复劳动
- 标准化新用户接待、绑角、发码、回访等流程
- 将人工客服从“重复问答”转向“高价值异常处理”
- 让管理端通过钉钉直接完成查询、跟进、干预和知识沉淀

## 2. 产品优势

相较于传统“人工客服 + 零散表格 + 群内通知”的模式，本产品有以下优势。

### 2.1 流程标准化

系统将游戏客服流程固化为 SOP 状态机，包含：

- 首次问候
- 收集区服和角色名
- 验证角色
- 绑定账号
- 自动发放兑换码
- 30 分钟回访
- 1 小时裂变引导
- 次日回访

这意味着服务质量不依赖个别客服的熟练度，降低了培训成本，也减少了流程遗漏。

### 2.2 响应效率高

玩家首次咨询时，系统可自动完成基础接待和标准发码。  
当玩家进入问答阶段，系统会先检索 OpenViking 知识库，再决定是否用 AI 组织答案或转人工。

效果上带来三点改善：

- 常见问题可以秒级响应
- 管理员无需反复查聊天记录和 FAQ 文档
- 复杂问题也能带着上下文进入人工处理

### 2.3 管理动作集中

管理端不需要登录多个后台，不需要直接看数据库，也不需要翻本地文件。  
管理员只需要在钉钉中对机器人下达自然语言指令，就能完成：

- 查看整体客服统计
- 查看客户列表和单个客户详情
- 查看待处理人工工单
- 主动给客户发消息
- 重置客户流程
- 关闭或恢复客户会话
- 向知识库新增人工确认过的标准问答

### 2.4 知识沉淀可复用

系统已接入 OpenViking 作为知识库底座，支持将 FAQ、活动规则、运营说明、经验问答沉淀为可检索资源。  
新补充的问答一旦写入知识库，后续同类问题就可以被命中并复用。

这意味着：

- 一个客服解决过的问题，后续可以被整个团队复用
- 培训材料不再只停留在文档里，而能进入在线服务能力
- 管理端可以边运营边积累组织知识

### 2.5 人工兜底明确

本系统不是“全自动黑盒”。当知识库置信度不足、玩家情况异常、业务规则不明确时，系统会生成待人工处理任务，由管理员通过钉钉接手。  
这保证了业务可控性，也降低了错误自动回复的风险。

## 3. 系统整体结构

当前版本建议按“双进程 + 多渠道”结构运行。

### 3.1 玩家侧服务进程

`nanobot.game_cs.service`

职责：

- 接收玩家消息
- 推进 SOP 流程
- 调用知识库和 AI
- 在需要时创建人工处理任务
- 对外暴露 webhook、admin、cron 接口

### 3.2 管理侧 Gateway 进程

`nanobot.cli.commands gateway`

职责：

- 承接钉钉消息
- 调用专用 `game_cs_*` 管理工具
- 将管理动作转发给 `game_cs` 的管理接口

### 3.3 钉钉管理通道

项目中已内置钉钉 Stream Mode 通道实现，位于 [dingtalk.py](F:\project\nanobot-customer-service\nanobot-game-customer-service\nanobot\channels\dingtalk.py)。

钉钉端的定位不是“单独后台页面”，而是“管理交互入口”。  
也就是说，管理员的主要操作发生在钉钉会话里，系统再将指令映射到后台 API。

### 3.4 OpenViking 知识库

知识库当前由嵌入式 OpenViking 提供支撑，默认本地目录为：

- [openviking_data](F:\project\nanobot-customer-service\nanobot-game-customer-service\.nanobot\openviking_data)

主要作用：

- 索引 FAQ、活动规则、运营资料
- 支持语义检索
- 支持人工新增标准问答
- 支持将长期有价值的交互沉淀为记忆

## 4. 典型业务闭环

从业务视角，本系统完整闭环如下。

1. 玩家发来消息。
2. 系统识别当前玩家是否处于首次接待、补资料、发码后跟进、知识问答等阶段。
3. 如果是标准流程，机器人自动处理。
4. 如果是知识型问题，系统优先查询知识库。
5. 如果知识库足够命中，则输出回答。
6. 如果问题异常或不确定，则转为人工待处理任务。
7. 管理员通过钉钉查看任务，并下达处理指令。
8. 管理员发现新的标准问答后，可通过钉钉将其沉淀到知识库。
9. 后续相似问题会优先由知识库命中，提高自动化率。

这个闭环是本产品最重要的管理价值：  
不是“自动回复一次”，而是“自动接待、人工补位、持续学习”。

## 5. 钉钉管理端使用前提

管理员在钉钉里能顺利使用系统，前提是以下条件已完成。

### 5.1 服务端已启动

需要至少启动两个进程：

```bash
set GAME_CS_SERVICE_TOKEN=replace-with-strong-token
set GAME_CS_AI_ENABLED=true
python -m nanobot.game_cs.service --host 127.0.0.1 --port 8011
```

```bash
set NANOBOT_GAME_CS_ADMIN_BASE_URL=http://127.0.0.1:8011
set NANOBOT_GAME_CS_ADMIN_TOKEN=replace-with-strong-token
python -m nanobot.cli.commands gateway -p 18790
```

其中：

- `NANOBOT_GAME_CS_ADMIN_TOKEN` 必须和 `GAME_CS_SERVICE_TOKEN` 相同
- `gateway` 才是管理员侧接触的服务
- `game_cs` 才是真正执行客服业务的服务

### 5.2 钉钉通道已启用

钉钉通道配置结构在 [schema.py](F:\project\nanobot-customer-service\nanobot-game-customer-service\nanobot\config\schema.py) 中定义，关键字段包括：

- `channels.dingtalk.enabled`
- `channels.dingtalk.client_id`
- `channels.dingtalk.client_secret`
- `channels.dingtalk.allow_from`

含义如下：

- `enabled`: 是否启用钉钉通道
- `client_id`: 钉钉应用 AppKey
- `client_secret`: 钉钉应用 AppSecret
- `allow_from`: 允许使用该机器人管理能力的员工 `staff_id`

### 5.3 管理员权限已加入 allow_from

管理员必须在钉钉允许名单中，否则虽然能接入钉钉机器人，但不会被允许真正进入消息处理链路。  
培训时要特别强调：机器人“能加上”和“有管理权限”是两件事。

### 5.4 管理模式已生效

当 `gateway` 检测到以下两个环境变量存在时，会自动进入 `game_cs` 管理模式：

- `NANOBOT_GAME_CS_ADMIN_BASE_URL`
- `NANOBOT_GAME_CS_ADMIN_TOKEN`

进入该模式后，Agent 会优先使用专用 `game_cs_*` 工具，而不是本地文件工具来回答管理问题。

## 6. 管理端能做什么

当前版本的钉钉管理端可以覆盖以下操作。

### 6.1 查询类能力

- 查看整体客服统计
- 查看最近客户列表
- 按 SOP 状态筛选客户
- 查询单个客户详情
- 查询单个客户最近消息
- 查看人工待处理工单

### 6.2 操作类能力

- 主动给客户发送管理员消息
- 重置某个客户的 SOP 流程
- 关闭客户的 AI 自动接待
- 恢复客户的 AI 自动接待
- 回复人工工单
- 导入现有知识库文件
- 新增一条人工确认的问答到知识库

### 6.3 知识管理能力

系统当前支持通过接口新增单条问答：

- `POST /kb/qa`

这意味着管理员不需要手工登录 OpenViking，也不需要手动编辑目录文件，就能把一条问答沉淀为知识库资源。

## 7. 管理员在钉钉中的完整操作流程

以下内容是培训管理员时建议采用的标准流程。

### 7.1 第一步：进入钉钉机器人会话

管理员在钉钉中找到接入好的机器人，通常以单聊方式进行管理。  
当前实现中，钉钉私聊的 `chat_id` 默认等于发送者 `staff_id`，因此每个管理员的会话天然可路由。

建议培训时先完成三项验证：

1. 机器人是否在线
2. 管理员是否在 allow list 中
3. 简单询问“查看当前客服统计”是否能拿到结果

### 7.2 第二步：先学会查询，不先学会操作

培训新管理员时，不建议上来就教“发消息”“关闭 AI 接待”这类动作。  
应该先让他们掌握查询链路，因为查询是理解系统状态的基础。

推荐训练顺序：

1. 查看统计
2. 列出客户
3. 查看某个客户详情
4. 查看人工工单

推荐指令示例：

- `查看当前客服统计`
- `列出最近 20 个客户`
- `列出处于 collecting_info 的客户`
- `查看客户 player_1001 详情`
- `查看客户 player_1001 最近消息`
- `查看待人工处理工单`

### 7.3 第三步：掌握人工干预

管理员在理解系统状态后，才进入人工干预类操作。  
这一阶段的重点是“先判断，再操作”，而不是见到问题就立刻发消息。

推荐操作顺序：

1. 看客户详情
2. 看最近消息
3. 确认是否需要人工回复
4. 再执行发消息、重置或关闭 AI 接待

推荐指令示例：

- `给客户 player_1001 发消息：请重新登录后再试`
- `重置客户 player_1001 的会话`
- `关闭客户 player_1001 的 AI 自动接待`
- `恢复客户 player_1001 的 AI 自动接待`

### 7.4 第四步：处理人工工单

人工工单通常发生在以下情形：

- 玩家问题超出知识库覆盖范围
- 游戏绑定或验证出现异常
- 用户表达不清但问题重要
- 业务规则尚未沉淀为标准答案

推荐培训动作：

1. 查看待处理工单
2. 确认 `query_id`、`user_id`、问题内容
3. 回复工单
4. 必要时继续跟踪客户详情

推荐指令示例：

- `查看待人工处理工单`
- `回复工单 42，用户 player_1001：请提供角色截图，我帮你继续处理`

### 7.5 第五步：沉淀知识库

这是管理端培训里最容易被忽略、但最有长期价值的一步。  
如果管理员只是回复工单，而不沉淀知识，那么系统永远只能依赖人工。

建议培训时强调以下原则：

- 重复出现的问题，必须进入知识库
- 人工确认过的标准话术，优先沉淀
- 活动类、发码类、规则类变化后，知识库也要同步更新

推荐自然语言用法：

- `新增知识库问答，问题：这个游戏有什么职业，答案：有法师、战士、牧师、魔法师`
- `新增 FAQ，问题：签到奖励怎么领，答案：进入活动页面点击每日签到即可领取`

这类指令最终会走到专用 `game_cs_add_kb_qa` 工具，再调用后台的 `/kb/qa` 接口。

## 8. 日常管理标准流程

为了便于培训管理端人员，建议把日常工作拆成四个固定时段。

### 8.1 班前检查

目标：确认系统是否健康，确保当天开班时机器人、知识库和管理通路可用。

检查项：

- 查看整体客服统计
- 确认服务在线
- 查看是否有遗留 pending 工单
- 确认当日兑换码是否已更新

推荐操作：

- `查看当前客服统计`
- `查看待人工处理工单`

### 8.2 值班中处理

目标：及时识别异常用户、人工接管高风险问题、保持玩家体验稳定。

处理顺序建议：

1. 先看工单
2. 再看高风险客户
3. 再做主动干预

高风险信号包括：

- 同一用户反复卡在 `collecting_info`
- 同一用户多次重置
- 绑定后仍反复投诉
- 某类问题短时间内集中出现

### 8.3 班中知识沉淀

目标：将“今天第一次出现、后续还会再出现”的问题尽快沉淀。

判断标准：

- 这个问题会不会再次发生
- 答案是否足够稳定
- 运营是否已经确认该答案可以对外统一使用

满足以上条件后，应当当天沉淀，不建议拖到周报再补。

### 8.4 班后复盘

目标：为第二天运营和培训提供依据。

建议输出：

- 今日总客户量
- 今日自动处理比例
- 今日人工处理量
- 今日新增 FAQ 数量
- 今日高频异常问题

虽然当前版本没有独立复盘报表页面，但管理员可以通过统计、客户列表和工单列表手工完成复盘。

## 9. 知识库管理规则

为了避免知识库变成“另一套混乱资料堆”，建议采用以下管理规则。

### 9.1 适合写入知识库的内容

- 稳定的游戏规则
- 常见 FAQ
- 标准操作说明
- 运营活动口径
- 已人工确认、可复用的标准答案

### 9.2 不适合直接写入知识库的内容

- 尚未确认的临时说法
- 高争议投诉处理结论
- 仅针对某个用户的个例判断
- 每小时都在变的临时运营话术

### 9.3 推荐新增知识库的判断标准

满足以下至少两项，就建议入库：

- 一周内重复出现两次以上
- 答案可以标准化
- 答案一旦标准化就能明显减少人工工作量

## 10. 培训讲师建议话术

如果你要培训管理端人员，建议按以下结构授课。

### 10.1 第一部分：先讲目标

不要一开始讲接口或命令。  
先让学员知道系统的目标是：

- 让机器人处理标准流程
- 让人工只处理复杂问题
- 让每次人工处理都尽量转化为未来可复用知识

### 10.2 第二部分：讲四类动作

将所有操作压缩成四类，学员更容易理解：

1. 查状态
2. 查用户
3. 做干预
4. 沉淀知识

### 10.3 第三部分：讲动作顺序

强调一个核心规则：

先查，再判，再操作。

不要直接下指令去改客户状态，而要先理解客户当前状态。

### 10.4 第四部分：讲知识沉淀责任

培训结束前必须强调：

- 人工回复不是终点
- 重复问题不沉淀，就会反复占用人工
- 知识库不是技术同学专属，运营和客服主管同样要参与维护

## 11. 常见场景演练

### 场景一：玩家无法绑定

处理流程：

1. 管理员查看客户详情
2. 检查 area_name、role_name 是否完整
3. 如果信息混乱，先重置会话
4. 如果信息正确但仍失败，转人工或联动游戏后端

推荐钉钉操作：

- `查看客户 player_1001 详情`
- `查看客户 player_1001 最近消息`
- `重置客户 player_1001 的会话`

### 场景二：玩家咨询常见规则

处理流程：

1. 先看系统是否已自动回复
2. 如工单仍出现，人工确认答案
3. 该问题若可复用，则立即加入知识库

推荐钉钉操作：

- `查看待人工处理工单`
- `回复工单 52，用户 player_1002：职业有法师、战士、牧师、魔法师`
- `新增知识库问答，问题：这个游戏有什么职业，答案：有法师、战士、牧师、魔法师`

### 场景三：玩家长时间无响应

处理流程：

1. 查看客户当前 SOP 状态
2. 判断是否已进入 follow-up 阶段
3. 如有必要由管理员主动补发消息

推荐钉钉操作：

- `查看客户 player_1003 详情`
- `给客户 player_1003 发消息：看到消息后请补充区服和角色名`

## 12. 风险与边界

培训中必须明确以下边界，避免管理员误解系统能力。

### 12.1 机器人不是最终裁决者

系统可以提高效率，但不能替代业务决策。  
涉及补偿、封禁、申诉、跨账号争议等高风险问题，仍应走人工决策。

### 12.2 知识库不是自动真理

知识库命中只能说明“内容相似”，不能代表“业务结论一定正确”。  
运营活动变化后，旧知识如果不更新，系统也会继续检索到旧答案。

### 12.3 发消息依赖 chat_id

管理员主动给客户发消息前，系统需要该客户已记录可用的渠道和会话标识。  
如果没有 `metadata.chat_id`，消息可能只能记入系统，不能实际送达。

### 12.4 管理员权限必须控制

钉钉通道必须使用 `allow_from` 限制管理员范围。  
否则任何接入机器人能力的人都可能尝试访问管理动作，这是不安全的。

## 13. 建议落地方式

如果你要把这套系统正式用于团队培训和日常运营，建议这样推进。

### 13.1 第一阶段：先跑通

- 启动 `game_cs`
- 启动 `gateway`
- 打通钉钉机器人
- 选 1 到 2 名管理员试运行

### 13.2 第二阶段：先控范围

- 只开放查询和工单回复
- 暂缓开放大规模主动外呼
- 先由主管审核知识库新增内容

### 13.3 第三阶段：形成制度

- 规定哪些问题必须入知识库
- 规定谁有权限新增 FAQ
- 规定班前检查和班后复盘动作

### 13.4 第四阶段：做指标

建议至少跟踪以下指标：

- 自动处理率
- 人工工单率
- 平均首响时间
- 高峰时段客服负载
- 知识库新增量
- 重复问题下降趋势

## 14. 当前版本可用于培训的结论

基于当前代码实现，本产品已经具备以下培训和运营条件：

- 已有面向玩家的完整 SOP 服务链路
- 已有钉钉管理通道
- 已有管理工具集合 `game_cs_*`
- 已有统计、客户管理、人工工单处理能力

如果后续还要进一步提升培训效果，建议下一步补两项：
